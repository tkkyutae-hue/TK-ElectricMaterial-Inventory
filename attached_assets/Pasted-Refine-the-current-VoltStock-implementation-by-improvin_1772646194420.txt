## 1. Inventory list pagination

The Inventory list currently shows too many items at once and requires endless scrolling.

### Required change

Add pagination to the Inventory table.

### Behavior

* place a page size control next to the search/filter area
* default page size should be **25 items per page**
* allow simple pagination controls at the bottom:

  * previous
  * next
  * page number(s) if practical

### Goal

Users should not have to endlessly scroll through the Inventory list.

## 5. Improve Inventory search matching

The current search is too strict.
Users should be able to search by partial word combinations.

### Example problem

Searching:

* `3/4"` works
* but `3/4" conduit` does not return `3/4" EMT Conduit`

### Required search behavior

Make search more flexible so results match if the query terms appear anywhere in the item data, even if not as one exact phrase.

### Search should match against:

* SKU
* item name
* size
* category
* grouped family / base item name if available

### Matching logic

Tokenize the search phrase and match by contained terms.
For example:

* `3/4 conduit` should match `3/4" EMT Conduit`
* `emt 3/4`
* `compression 1`
* `mc 12/2`
* `ground lug`
  should all work naturally.

The goal is a practical fuzzy/contains-style search, not exact full-string matching only.

## 6. Transaction History improvements

The current Log Movement form lets users enter From location on Receive, but that value does not show in the actual transaction log.
Also, there needs to be a Project field.

### Required fixes

#### A. Show From / To correctly in transaction history

* if a movement includes a From location, display it in the transaction history table
* if a movement includes a To location, display it too
* make sure the transaction log actually persists and shows those values correctly

#### B. Add Project field

In the Log Movement form, add a Project field where practical.

### Project field behavior

* allow selecting an existing project
* save the project reference with the movement record
* display the selected project in the Transaction History table

### Movement types

For each movement type, show relevant fields:

* Receive → From location should still be supported if that is the current business rule
* Issue → From, To, Project as relevant
* Return → From, To, Project as relevant
* Transfer → From and To
* keep logic clean and only show necessary fields

### Final transaction table

Make sure the transaction history table can display:

* Date & Time
* Type
* Item
* Qty
* From
* To
* Project
* Note

If values exist, they must actually show.
