TASK GOAL
Fix the request editing flow so that when a user edits an existing request, goes to Inventory, adds/removes items, and submits again, the app updates the existing request instead of creating a brand new request.

CURRENT BUG
Right now the flow behaves like this:
1. User edits existing request (example: REQ-0007)
2. User goes to Inventory from request edit flow
3. User modifies cart by adding/removing items
4. User presses Submit Request
5. The app creates a NEW request instead of updating REQ-0007

This is wrong.

CORRECT BEHAVIOR
If the user is currently editing an existing request, Inventory/cart must remain in "edit existing request" mode.

That means:
- the request number stays the same
- submit action should update the existing request
- no new request should be created
- the UI should clearly indicate the user is editing an existing request

FILES TO FOCUS ON
Inspect the repo first and update only the relevant files for:
- Field request edit flow

Introduce a clear editing context in the cart/request flow.

The cart/request state should be able to distinguish:
- new request mode
- editing existing request mode

Recommended shape (adapt to the repo as needed):
- mode: "new" | "edit-request"
- editingRequestId
- requestType
- project
- requester
- notes
- items

Requirements:
1. When entering Inventory from an existing request edit flow, set the cart into edit-request mode
2. Preserve the original request id in that editing context
3. Preserve the current editable request payload in the cart state
4. Do NOT treat this as a new blank request cart
