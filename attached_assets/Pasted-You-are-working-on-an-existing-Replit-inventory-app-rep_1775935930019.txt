TASK GOAL
Implement clear role-based request permissions with the following business rules:

PERMISSION RULES
- viewer: cannot create or modify requests
- staff and above: can create requests
- staff and above: can edit/cancel their OWN requests only while status is `requested`
- manager/admin and above: can edit requests, move them to `preparing`, `ready`, `completed`, and cancel/delete according to current safe request design
- manager/admin should be able to manage requests created by others

IMPORTANT BUSINESS INTENT
The requester should be able to fix or update their own request before warehouse/admin has started working on it.
Once a request leaves `requested` and enters warehouse-side workflow, requester editing should stop.

FILES TO FOCUS ON
Inspect the repo first and update only the relevant files for:
- request routes/handlers

PART 1 — ROLE MATRIX
Implement the following server-enforced permission rules:

A. CREATE REQUEST
- staff, manager, admin: allowed
- viewer: not allowed

B. REQUESTER SELF-EDIT
- requester can edit their own request only when:
  - status = requested
  - role is staff or higher

C. REQUESTER SELF-CANCEL
- requester can cancel their own request only when:
  - status = requested
  - role is staff or higher

D. MANAGER/ADMIN REQUEST CONTROL
- manager/admin can:
  - edit requests
  - set status to preparing
  - set status to ready
  - set status to completed
  - cancel/delete according to the repo’s safe request model

Important:
- Enforce these rules on the server, not just in the UI
- Do NOT rely only on hidden buttons
